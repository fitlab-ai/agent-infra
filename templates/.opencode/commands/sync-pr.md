# Sync PR Command

## Description

Sync a task's progress summary to the corresponding Pull Request's comment section, helping reviewers understand development progress and key decisions.

## Execution Flow

### 1. Verify Task Exists

Check if the task file exists:
- Check `.ai-workspace/active/{task-id}/task.md`
- If not found, check `completed/` and `blocked/` directories
- If none found, prompt that the task does not exist

### 2. Read Task Information

Get from the task file:
- PR number (`pr_number` field)
- Task title and description
- Current step (`current_step`)
- Task status (`status`)
- Created and updated timestamps
- Related Issue (`issue_number`, if any)

### 3. Read Context Files

Note: `{task-id}` format is `TASK-{yyyyMMdd-HHmmss}`, e.g., `TASK-20260205-202013`

Check and read the following files (if they exist):
- `.ai-workspace/{status}/{task-id}/analysis.md` - Requirement analysis
- `.ai-workspace/{status}/{task-id}/plan.md` - Technical plan
- `.ai-workspace/{status}/{task-id}/implementation.md` - Implementation report
- `.ai-workspace/{status}/{task-id}/review.md` - Review report

### 4. Generate Progress Summary

Generate a clear progress summary based on the current status:

**Basic format**:
```markdown
## 🤖 Development Progress Update

**Task ID**: {task-id}
**Updated**: {current time}
**Current Status**: {status description}

### ✅ Completed

- [x] Requirement Analysis - {completion time}
  - {Core summary points 1-2 items}
- [x] Technical Design - {completion time}
  - {Solution selection and key decisions 1-2 items}
- [x] Code Implementation - {completion time}
  - Modified files: {count}
  - New tests: {count}
- [ ] Code Review (in progress)
- [ ] Final Merge

### 📋 Current Progress

{Detailed description of current step}

### 🎯 Next Steps

{Next step plan}

### 📊 Technical Highlights

{Key technical decisions and implementation details for reviewer understanding}

### 📂 Related Documents

- Task file: `.ai-workspace/active/{task-id}/task.md`
- Requirement analysis: `.ai-workspace/active/{task-id}/analysis.md`
- Technical plan: `.ai-workspace/active/{task-id}/plan.md`
- Implementation report: `.ai-workspace/active/{task-id}/implementation.md`

---
*Auto-generated by Claude Code - [Task Management System](../.agents/README.md)*
```

**Summary principles**:
- **Reviewer-oriented**: Highlight technical decisions and implementation points
- **Concise and clear**: Extract only core points for each phase
- **Logical flow**: Show progress following the development workflow
- **Review-friendly**: Explain the reasons and impact of key changes

### 5. Sync to PR

Use `gh` command to post the summary to the PR comment:

```bash
gh pr comment {pr-number} --body "$(cat <<'EOF'
{generated progress summary}
EOF
)"
```

### 6. Update Task Status

Record the sync time in the task file:
- Add or update `last_synced_to_pr_at` field
- Record the PR comment link (if returned by gh)

### 7. Inform User

Output format:
```
Task progress synced to PR #{pr-number}

**Synced Content**:
- Completed steps: {count}
- Current status: {status}
- Next step: {next step description}

**View Link**:
https://github.com/{owner}/{repo}/pull/{pr-number}
```

## Parameters

- `<task-id>`: Task ID, format TASK-{yyyyMMdd-HHmmss} (required)

## Usage Example

```bash
# Sync task progress to corresponding PR
/sync-pr TASK-20251227-104654
```

## Notes

1. **PR Number Must Exist**:
   - Task file must have a `pr_number` field
   - If not, prompt user to manually specify or update the task file

2. **Summary Generation Principles**:
   - Written for code reviewers
   - Highlight technical decisions and implementation points
   - Explain the reasons for key changes
   - Avoid excessive technical details

3. **Sync Timing**:
   - When code implementation is complete and ready for review
   - After processing review feedback
   - When there are significant progress or decision changes
   - Progress reminder when PR has been waiting for review for a long time

4. **Comment Format**:
   - Use Markdown formatting
   - Use emoji for readability
   - Include timestamps
   - Add Claude Code signature

5. **Avoid Frequent Syncing**:
   - Don't sync at every small change
   - Recommend syncing after completing important phases
   - Avoid generating too many comments

6. **Relationship with Issue Sync**:
   - `/sync-issue` is oriented toward project managers and stakeholders
   - `/sync-pr` is oriented toward code reviewers and developers
   - The two have different content focus areas

## Related Commands

- `/create-pr` - Create Pull Request
- `/sync-issue <task-id>` - Sync progress to Issue
- `/review-task` - Code review
- `/commit` - Commit code

## Error Handling

- Task not found: Prompt "Task {task-id} not found, please check the task ID"
- Missing PR number: Prompt "Task file is missing the pr_number field"
- PR not found: Prompt "Pull Request #{pr-number} does not exist"
- PR already closed: Prompt "Pull Request #{pr-number} is already closed"
- gh command failed: Prompt "Sync failed, please check if GitHub CLI is authenticated"
- Network error: Prompt "Network connection failed, please try again later"

## Best Practices

1. **Sync immediately after development completion**:
   ```bash
   # Recommended workflow
   /implement-task TASK-xxx    # Implement feature
   /sync-pr TASK-xxx           # Sync progress, request review
   ```

2. **Sync again after handling review feedback**:
   ```bash
   # After fixing review issues
   git commit -m "fix: address review comments"
   /sync-pr TASK-xxx           # Inform reviewer issues are addressed
   ```

3. **Reminder for long waits**:
   - If PR has been waiting for review for over 2 days, sync a progress reminder
   - Indicate urgency level and blocking situation

4. **Team collaboration**:
   - @mention relevant reviewers in the progress summary
   - Indicate areas that need special attention
   - Highlight important technical decisions
